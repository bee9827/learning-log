# 레벨2 학습·설계 정리 — 룸이스케이프 예약 시스템

프로젝트: `spring-roomescape` (member → auth → waiting), **JDBC 기반**.

이번 레벨2의 기술적 고민은 **트랜잭션(ACID)을 축으로** 이어졌다. **격리성(I)**은 *동시성·락*으로, **원자성(A)**은 *트랜잭션 경계·아웃박스*로, **일관성(C, 비즈니스 규칙)**은 *도메인 모델링*으로 풀렸고 — 그 경계가 실제로 지켜지는지는 *테스트*로 증명했다. 이 전체를 **나선형 학습법**으로 한 repo에 쌓아 재방문했다.
공통적으로, 정답을 외우기보다 **"왜 다른 선택지가 안 되는지"를 따져서** 결정했다.

---

## 1. 트랜잭션 — 격리성(동시성)과 원자성(경계), 두 면

**격리성(I) — 동시성은 한 종류가 아니었다.** 같은 '동시성'인데 셋이 *각각 다른 도구*를 요구했고, 그걸 구분한 게 핵심.
- **INSERT 중복 예약 경쟁**: 비관/낙관 락을 떠올렸지만 *잠글 행이 없어* INSERT엔 락이 안 먹힌다 → **DB unique 제약**이 근본. soft delete와 충돌해 `deleted_at` + `UNIQUE(…, deleted_at)`로, *NULL은 유니크에서 제외*되는 성질을 이용해 "활성 예약만 유일"을 만듦.
- **예약 수정 UPDATE 충돌**: 경합이 드물어 **낙관적 락**. JDBC라 `@Version` 대신 `version` 컬럼 + `WHERE id=? AND version=?`, 영향 행 0이면 `409`.
- **대기 순번(rank) read-then-write 경쟁**: 비어 있을 수 있는 큐 대신 **항상 존재하는 부모(예약) 행을 `FOR UPDATE`로 잠가 직렬화**. waiting UNIQUE는 코드 경로가 빠질 때만 작동하는 "잠든 백스톱".
- → **무엇을 경쟁하나(INSERT/UPDATE/read-then-write) + 경합 빈도**가 도구를 결정한다.

**원자성(A) — 트랜잭션 경계를 어디서 끊을까 (아웃박스).** 예약 취소 시 ①취소(반드시 성공) ②다음 대기자 승격(재시도 가능)이 필요. 묶으면 ②실패가 ①을 막고, 떼면 ②가 증발.
- **결정**: 승격 "의도"를 취소와 *같은 트랜잭션*으로 DB(outbox)에 기록 → `@Scheduled` 워커가 재시도 실행. 강한 일관성 → **최종 일관성**.
- **대가**: 폴링 간격만큼 "빈 창"·새치기. 워커 중복 실행은 *"활성 예약 있으면 승격 안 함" + UNIQUE 백스톱*으로 **멱등성** 확보.
- **원리**: 원자성은 한 DB까지. 외부/분리 작업은 "외부 행동을 DB 쓰기로 바꿔" 트랜잭션에 넣고, 실제 실행은 비동기+재시도+멱등성으로 받친다.

## 2. 복잡한 모델링 — 비즈니스 규칙(C)을 어디에 둘까

Waiting 규칙이 Service에 퍼져 도메인이 빈 껍데기였고, Reservation과 공통 필드가 많았다.
- **공통 개념 추출**: "Waiting이 Reservation을 통째 소유한다" 가설을 *라이프사이클*로 탈락(예약이 사라지면 대기가 승격 → 생명주기 엇갈림) → 공통 필드를 **Slot 값 객체**로 뽑고 판단(`isPast`)도 거기 둠.
- **규칙 위치 기준 "행 간 불변식"**: 여러 행을 봐야 판단되는 규칙(예약 존재·중복 대기)은 객체 하나가 자기만 알아 판단 불가 → Repository+락 자리(↑ I와 연결). 단일 객체로 판단되는 것(동일 사용자)만 도메인으로.
- **일급 컬렉션**: rank 계산을 `Waitings`가 책임. 코드 리뷰 14개 finding 중 6개가 *"데이터를 가진 객체가 책임진다"* 한 원칙으로 묶였고, 한 점을 풀자 같이 풀렸다.

## 3. 테스트 — 웹 계층부터 트랜잭션 경계 증명까지

- **Controller 테스트**: 상태코드(201/400/409)를 `RestAssuredMockMvc`로(given/when/then 가독성). Service 변경에 흔들리던 예외 케이스를 단위 테스트로 **분리**해 Controller 중복 제거.
- **트랜잭션 경계 롤백 테스트** (테스트↔트랜잭션 연결): `@MockitoSpyBean`으로 *진짜 DAO인데 `delete`만 실패*를 주입 → `cancel()`의 `@Transactional`이 원자적으로 롤백하는지 검증. **함정**: 테스트가 트랜잭션을 쥐면 `cancel`이 거기 *join*해 물리 롤백이 미뤄지고, 검증이 `CANCELED`를 봐버림 → *"독립 경계를 관찰하려면 테스트가 트랜잭션을 쥐면 안 된다"* → `@SpringBootTest` 선택. (트랜잭션 = 커넥션이 쓰레드에 바인딩되는 방식이라는 메커니즘과 직결.)

## 4. 학습법 — 나선형 학습 + 중앙 로그

흩어진 학습을 한 repo(로그 32개)로 모아 같은 키워드를 **다른 맥락·깊이로 재방문**했다 (낙관적 락 *개념*→*JDBC 구현*, 갭락 *개념*→*락 순서*→*데드락*). 매번 "왜 안 되는지"를 스스로 탈락시키며 좁혔고, 사이클마다 **학습 방식 자체에 대한 회고("다음에 바꿀 것")**를 남겨 방법을 고쳐갔다. 다음에 팔 키워드(i+1)는 BACKLOG로 관리한다.
