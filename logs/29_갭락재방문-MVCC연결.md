# 학습 로그 #29

**날짜**: 2026-06-09
**학습 범위**: 갭락 재방문 — MVCC와의 관계, SELECT FOR UPDATE 스냅샷 우회

분류: 락-동시성 (재방문: #17 갭락-NextKeyLock, #12 UndoLog-MVCC)

## 1. 이전 대비 깊어진 부분

#17에서는 갭락의 구조(무엇인지, 어디에 걸리는지)를 다뤘다.
오늘은 "왜 MVCC로 팬텀 리드를 막을 수 없는가" — MVCC와 갭락의 역할 분담까지 연결했다.

## 2. 핵심 흐름

### 갭락이 필요한 이유

- Repeatable Read는 팬텀 리드를 막아야 한다
- 없는 행엔 X락을 걸 수 없다
- → 갭락으로 인덱스 빈 공간에 삽입을 막는다

### MVCC로 팬텀 리드를 못 막는 케이스

- 일반 `SELECT`는 MVCC 스냅샷을 읽는다 → 팬텀 리드 안 일어남
- `SELECT FOR UPDATE`는 X락을 걸어야 하므로 **현재 최신 커밋 데이터**를 읽는다 (스냅샷 우회)
- → 트랜잭션 사이에 다른 트랜잭션이 삽입한 행이 보일 수 있다
- → 여기서 갭락이 필요하다

### 격리 수준별 스냅샷 기준

| 격리 수준 | 스냅샷 시점 |
|---|---|
| Read Committed | 쿼리마다 새로 찍음 |
| Repeatable Read | 트랜잭션 시작 시 한 번 |

### 인덱스와 갭락 범위

- 갭락은 B-Tree 인덱스 리프 노드 사이를 기준으로 걸린다
- 인덱스 없는 컬럼에 범위 쿼리 → 풀 테이블 스캔 → 테이블 전체에 갭락
- → 다른 트랜잭션의 모든 삽입이 막힌다 (사실상 테이블 수준 잠금)

## 3. 스스로 발견한 것

- 갭락끼리 충돌하지 않는다 → 삭제/수정은 갭락이 아닌 X락이 막는다
- `SELECT FOR UPDATE`가 스냅샷을 우회하는 이유: X락을 잡는 시점의 최신 데이터를 봐야 하기 때문
- MVCC의 스냅샷은 "어떤 버전까지 볼지"의 포인터 — 새 테이블을 만드는 게 아님

## 4. 아쉬운 점 (학습 방식)

- `SELECT FOR UPDATE`가 스냅샷을 우회한다는 걸 시나리오 없이는 감이 안 왔다 — 개념만으론 와닿지 않는 케이스

## 5. 다음 사이클 키워드

```
✅ B-Tree에서 갭락이 물리적으로 어떻게 구현되는가
   → 맥락: 인덱스 리프 노드 사이를 어떻게 "잠그는지" — 구조 레벨에서
   → 종류: 흐름 파악 (#17에서 예고했지만 아직 미탐구)

✅ SELECT FOR UPDATE의 스냅샷 우회 — 내부적으로 어떻게 동작하는가
   → 맥락: 오늘 "현재 최신 데이터를 읽는다"는 건 알았지만 어떻게 구현되는지는 모름
   → 종류: 흐름 파악
```
