# 학습 로그 #30

**날짜**: 2026-06-15
**학습 범위**: ACID를 "지향점"으로 다시 세우고, 흩어진 DB·트랜잭션 로그(03·05·06·08·12·13·17·26·28)를 하나의 인과 사슬로 꿰는 메타 복습.

분류: 메타-복습 (재방문: 위 로그 전부 / 근본 재정렬)

## 1. 이 사이클의 성격

기존 로그들은 락·격리수준·MVCC·아웃박스를 *각각* 깊게 팠지만, "**왜 이게 다 필요한가**"의 뿌리(ACID)를 못 짚고 있었다. 본인이 "근본을 못 짚었다, DB의 지향점을 알고 싶다"고 자각 → bottom-up 퀴즈 대신 **학습자가 먼저 풀어내고 코치가 보완·연결**하는 방식으로 진행.

## 2. 핵심 사슬 (시작점 ACID → 끝점 unique, 한 바퀴)

> **목표(ACID) → I만 다이얼 → 정확성↔동시성 저울 → MVCC로 읽기 싸게 → 쓰기 충돌(Lost Update)은 조용해서 위험 → 락으로 막되 경합 빈도로 낙관/비관 선택 → unique 백스톱 → A의 사정거리는 한 DB까지 → 외부(결제·알림)는 못 묶음 → 아웃박스/Saga → 재시도 → 멱등성 → 멱등성도 결국 unique로 강제.**

### DB의 지향점

- 여러 사람이 동시에 쓰는데 각자는 "나 혼자 쓰는 것처럼" 느끼게 하고, 정전에도 약속한 결과는 살아남게 한다.
- *동시에*(throughput)와 *혼자처럼*(correctness)이 싸운다. 이 긴장의 계약서 = ACID, 구현 기계 = 락·MVCC·격리수준.

### I = 유일하게 양보 가능한 다이얼

- A·D는 거의 안 양보, C는 DB제약+애플리케이션 양쪽, **I만 격리수준으로 조절 가능**.
- I를 세게 = 락 많이/오래 = 블락 = 느림. 약하게 = 동시성↑ = 이상현상(dirty/non-repeatable/phantom) 위험. → "정확성 ↔ 동시성" 저울.
- 낙관적 락 = 경합 드묾에 베팅(평소 락 0, 충돌 시만 예외→재시도). 비관적 락 = 경합 잦음 대비(미리 줄 세움, 재시도 폭주보다 블락이 쌈). **가르는 축 = 경합 빈도.**
- 예약(인기 슬롯 = 경합 높음) → 비관적 락 + unique 백스톱.

### unique = 선언적 최종 방어 (defense in depth)

- 락 = 절차적·협조적 방어("이 코드 경로를 지나는 트랜잭션끼리만" 줄 세움 → 락 안 타면 무력).
- unique = 데이터에 박힌 규칙 → 미래의 개발자가 락을 깜빡한 쿼리를 추가해도 DB가 마지막에 거부. **사람 실수와 무관하게 안 무너지는 방어.**

### A의 사정거리 = 한 자원(DB)까지

- 같이 묶여야 하는 일(예약+결제)은 한 트랜잭션. 하지만 **외부 시스템에 손이 닿는 순간 롤백 불가**(보낸 이메일·완료된 PG결제는 못 되돌림).
- 아웃박스: "알림 발송"이라는 외부 행동을 "알림 보내라는 행 INSERT"라는 **DB 쓰기로 바꿔치기** → 예약·결제·아웃박스 행이 같은 DB라 원자적, 실제 발송은 Worker가 재시도(최종 일관성).
- 결제도 외부 PG라면 같은 문제 → 승인(authorization)/매입(capture) 2단계 + 메시지·재시도 = **Saga 패턴**.

### 멱등성 = 재시도의 짝, 그리고 unique로의 회귀

- "될 때까지 재시도"는 함정: 성공했는데 ack 유실 → 또 보냄 → **이중 결제**.
- 멱등성 키(주문당 고유 번호)로 중복 차단. 그 "같은 키는 한 번만"을 실제로 강제하는 장치 = **멱등성 키 컬럼의 UNIQUE**. → 사슬이 unique로 한 바퀴 닫힘.

## 3. 스스로 발견/교정한 것

- I만 다이얼이라는 것, 낙관/비관을 "경합 빈도"라는 단일 축으로 정리한 것 — 본인 인출.
- 결제가 외부 PG면 예약과 원자적으로 못 묶인다는 걸 스스로 도출 → 아웃박스/Saga 해법까지 재발명.
- 멱등성이 재시도의 필연적 짝임을 본인이 연결.

## 4. 아쉬운 점 (학습 방식)

- 마무리 자기요약에서 **D를 "Duration"으로 오기**, **C의 애플리케이션/DB 경계가 흐릿**한 게 드러났다. 가장 값진 신호(인출 시점에 약점이 정확히 노출됨)였는데, 이번엔 "다음 키워드"로 미뤘다 → 교정이 지연되고 오개념이 한 번 더 굳을 위험.

## 5. 다음 타임에 바꿀 것

**유지할 것**

- "학습자가 먼저 풀고 코치가 보완·연결" 모드 — 복습 단계에선 퀴즈가 흐름을 끊는다. 이번 판단이 정확했다.
- 학습자의 자기교정/메타인지("지금 학습모드인지", "I만 봤으니 A도 보자")를 코치가 가로채지 않고 방향타를 쥐게 둔 것.

**바꿀 것 (딱 하나)** — *학습법 에이전트 검증 반영*

마무리 인출에서 드러난 오류·흐릿함을 "다음 키워드"로 미루지 말고, **그 자리에서 한 박자 더 돌려 본인이 교정하게 한다.**
- **Why**: 인출 시점에 노출된 약점(D=Duration, C 경계)은 가장 값진 신호다. 키워드로 넘기면 교정이 다음 세션으로 지연되고 오개념이 굳는다. 인출 비중이 이미 높은 컴포트존이므로, 더 풀게 하기보다 *드러난 약점을 즉석에 닫는* 게 다음 성장 방향.
- **How**: 마무리 요약 직후 정답을 주지 말고 단 한 번의 대조 질문으로 되돌린다. (예: "정전에도 살아남는 성질이면 'Dur-'가 durable에 가까울까 duration에 가까울까?") 막히면 그때 키워드로 넘긴다(현 방식이 안전망). **한 박자만.**
- **이번 사이클에 즉시 적용·검증됨**: D="Duration" 오기를 그 자리에서 대조 질문 → 본인이 "Durability, 커밋 시점부터 영속" 으로 교정 성공. 사용자가 "찝찝하게 넘기면 잘못된 사실을 반복하는데 지금 짚으니 교정된다"고 효과를 명시. → **다음 사이클엔 "유지할 것"으로 승격 후보.**

### 보너스 — backstop 용어의 그림 (마무리 인출에서 추가로 닫음)

- backstop = 야구 포수 뒤 그물. 평소엔 0, 앞단(포수=락)이 놓친 공이 튀어나올 때만 막는다 → "평소 비용 0, 실패할 때만 값을 한다"가 defense in depth의 정서.
- **앱은 uniqueness를 원자적으로 못 지킨다**: "조회해서 없으면 INSERT"는 조회·INSERT 사이 race window가 있어 둘 다 통과(Lost Update와 동형). "한 번만"을 진짜 원자적으로 강제할 수 있는 곳은 DB unique 한 곳뿐 → 백스톱이 *DB에* 있어야 하는 이유. 사슬의 시작(동시성 race)과 끝(unique)이 여기서 다시 맞물림.
