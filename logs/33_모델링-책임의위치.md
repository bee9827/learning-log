# 학습 로그 #33

**날짜**: 2026-06-15
**학습 범위**: 복잡한 모델링(객체지향·도메인)의 지향점 — "책임을 어디에 둘까". 정보 전문가/캡슐화, 묶기↔나누기(역할 기준), 행 간 불변식 → 트랜잭션 용접.

분류: 도메인설계 (재방문: #23 Slot 값 객체, #24 규칙 도메인 이전, #25 조회 단순화 — *직접 인출로 한 줄 꿰기*)

## 1. 지향점 (뿌리 질문)

> "로직(책임)을 어디에 둘 것인가."

- **실패 모드**: 도메인에 있어야 할 로직이 Service/Dao로 새어 비슷한 게 흩어짐 → **빈 껍데기 도메인 = 절차지향**.
- **원리**: 데이터를 가진 객체가 행위도 가진다 = **정보 전문가(Information Expert) + 캡슐화 + Tell-Don't-Ask**. (본인 인출 키워드: 정보 전문가, 캡슐화.)

## 2. 저울 ① — 묶기 ↔ 나누기 (기준 = *역할/행위*, 데이터 모양 아님)

- Reservation/Waiting 메서드가 비슷 → 상속을 떠올림. 그러나 *라이프사이클이 다르고* is-a가 아님 → **상속 거부, 공통은 Slot(has-a)으로 컴포지션**.
- Slot+member를 한 번 더 묶을까? → 데이터는 같아도 **역할이 다름**(예약자 vs 대기자) → 안 합침.
- 결론: **비슷하다고 다 묶지 않는다 — 데이터가 같아도 역할이 다르면 나눈다.**

## 3. 저울 ② — 도메인 ↔ 협력 (기준 = 행 간 불변식 + 컬렉션 크기)

- 단일 객체로 판단 가능 → **도메인**.
- 여러 행과 협력 필요(**행 간 불변식**: "이미 예약 있나", "이미 대기했나") → 도메인 혼자 못 풂 → **Service**.
  - 컬렉션 작음(Waitings ≤ 5, 도메인 특성) → **일급 컬렉션**으로 도메인에 올림.
  - 컬렉션 큼(Reservations 전체) → **성능** 때문에 Service + DB 조회.
- 원칙(도메인 우선)을 *도메인 지식(방탈출 대기 5명 제한)*으로 조절한 게 핵심.

## 4. 트랜잭션과의 용접점 (throughline)

- "확인 → 없으면 INSERT"에서 **확인과 INSERT 사이의 race = INSERT 경쟁**. 즉 **모델링의 한계(행 간 불변식을 Service로 뺌)가 곧 트랜잭션(동시성)의 출발점**.
- 도메인은 자기 안에서만 원자적. 여러 행에 걸친 규칙은 **락/unique(I)**가 받친다. → 모델링(C) ↔ 트랜잭션(I) 연결.
