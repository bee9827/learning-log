# 학습 로그 #31

**날짜**: 2026-06-15
**학습 범위**: Saga 패턴 — 보상 트랜잭션(compensating transaction), Outbox와의 관계, rollback vs compensation.

분류: 비동기-이벤트 / 분산 트랜잭션 (재방문: #28 아웃박스, #30 ACID 메타)

## 1. 계기

log_30 사슬에서 "아웃박스/Saga"를 흘렸는데 Saga를 몰라 **같은 흐름에서 즉석 재방문**. (BACKLOG #30에서 막 연 키워드를 바로 당겨씀 — 사슬에서 삐져나온 i+1을 그 자리에서 소화.) 본인이 오늘 이미 Saga를 *재발명*했음(결제 확인→예약→결제 메시지→재시도→멱등성)을 짚고, 이름과 빈칸 하나(보상)를 채우는 방식으로 진행.

## 2. 핵심

### Saga = 여러 장(chapter)으로 이어지는 긴 이야기

- 문제: 큰 ACID 트랜잭션으로 **못 묶는** 작업들(외부/분산)의 일관성을 어떻게 지키나.
- 모양: 하나의 큰 트랜잭션 대신 → **작은 로컬 트랜잭션들을 순서대로** + **메시지로 바통**을 넘김. 각 장은 자기 자리에서 자기 일을 커밋.

### Outbox와의 관계 (자주 헷갈리는 지점)

- **Saga = 전체 줄거리** (로컬 트랜잭션들의 순서).
- **Outbox = 장↔장 바통(메시지)을 안 떨어뜨리는 배관** — "내 로컬 커밋"과 "다음 장 부르는 메시지 발송"을 원자적으로 묶어 dual-write 문제 해결.
- → Outbox는 Saga의 **부품 하나**지 Saga 전체가 아니다. (본인 첫 인출은 "Outbox가 Saga 구현 방법" — 절반 맞음, 부품임을 교정.)

### 실패 처리 — 두 갈래

- **일시적 실패** → 재시도 (forward recovery, 앞으로 밀어 성공시키기).
- **영구적 실패**(재시도 소진) → **보상 트랜잭션** (backward recovery, 되돌리기).
- 타이밍 구분: **아직 안 터진 장**은 outbox 메시지를 안 쓰면 끝(보상 불필요). **이미 커밋된 장**(결제 완료)은 보상으로만 되돌릴 수 있다.

### 보상 트랜잭션의 성질 (핵심)

- 롤백이 **아니라** "반대 행동을 하는 새 장"을 이야기에 덧붙이는 것. 결제→보상=환불, 예약→보상=예약취소.
- 흔적이 **남는** semantic undo: 환불은 결제 기록을 지우는 게 아니라 반대 거래를 추가. 장부엔 결제 줄 + 환불 줄 둘 다 남는다.

## 3. rollback vs compensation (본인 인출로 도출)

| | Rollback (ACID의 A) | 보상 (Saga) |
|---|---|---|
| 시간 | 되감는다(없던 일로) | 못 되감음 → 앞으로 나아가 반대 거래 추가 |
| 흔적 | 0 | 양쪽 다 장부에 남음 |
| 가능 이유 | 한 자원·커밋 전 | 외부·커밋 후라 되돌리기 불가 → 덧쓰기 |
| 결과 상태 | 원래 상태 | 비즈니스적으로 동등한 상태 |

## 4. ACID 척추와의 연결 (payoff)

- Saga = **A(원자성)를 한 DB 너머 여러 시스템으로 확장**한 것. 단 공짜가 아니다 — **되감기(rollback) 능력을 내주고 도달 범위(여러 시스템)를 얻는** 트레이드오프.
- log_30의 "A의 사정거리는 한 DB까지"라는 벽을, Saga는 *뚫는 게 아니라 우회*한다. (강한 원자성 ↔ 분산 도달.)
