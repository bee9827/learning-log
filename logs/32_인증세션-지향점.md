# 학습 로그 #32

**날짜**: 2026-06-15
**학습 범위**: 인증 & 세션의 지향점 — stateless HTTP 위에 "신원(상태)"을 얹기. 세션(서버 보관) vs JWT(클라이언트 보관), 확장성↔무효화 트레이드오프.

분류: 인증-세션 (재방문: #16 세션vs토큰, #18 동시로그인·JWT무효화, #19 JSESSIONID, #22 JWT구조)

## 1. 지향점 (뿌리 질문)

> "HTTP는 *상태가 없는데*, 어떻게 '이 사람 로그인했음'을 기억하지?"

- HTTP stateless = 매 요청 독립, 서버가 요청 사이 아무것도 기억 안 함. **이유 = 확장성·단순성**(아무 서버나 아무 요청 처리 가능).
- ※ 교정: "유저가 바꿀 수 있어서 stateless"는 어긋남. 그건 *클라이언트에 상태를 저장할 때의 보안 문제*이고 — JWT에서 정확히 다시 만난다(주머니에 넣어둔 통찰을 JWT에서 회수).
- 로그인 = stateless 위에 "신원이라는 상태"를 얹는 일. **그 상태를 어디 두느냐**가 카테고리 전체의 갈림.

## 2. 세션 = 서버가 보관 (stateful)

- 로그인 → 서버 세션에 상태 저장 + **JSESSIONID**(의미 없는 난수 = 참조 열쇠) 발급 → 쿠키로 전달.
- 브라우저가 쿠키 자동 관리 → 매 요청 자동 첨부 → 서버가 ID로 금고 조회.
- **JSESSIONID는 내용물이 아니라 참조(열쇠).** 유저는 열쇠만 들어서 "나 admin"이라 위조 불가. (열쇠 훔치는 *세션 하이재킹*은 별개 문제.)

## 3. JWT = 클라이언트가 보관 (stateless)

- 구조: **Header**(알고리즘) . **Payload**(권한 등 실제 정보) . **Signature**(무결성 검증).
- ★ 핵심 교정: **Payload는 암호화가 아니다.** Base64 인코딩이라 누구나 열어 읽음(jwt.io). JWT가 지키는 건 기밀성(비밀)이 아니라 **무결성(위조 방지)**.
- **Signature** = 알고리즘으로 `(Header.Payload)` + **서버만 아는 secret**을 돌려 만든 값. 검증 = 받은 내용으로 **재계산 후 비교**. Payload 한 글자만 바뀌어도 서명이 전부 바뀜(해시 눈사태 효과). 공격자는 **secret이 없어** 유효한 서명을 못 만듦 → 위조 즉시 들통.
- **payoff**: 서버는 secret 하나면 어떤 토큰이든 검증 → 유저별 상태 저장 불필요 = **stateless**. (세션은 유저마다 서버 금고가 필요했음.)

## 4. 지향점을 닫는 저울 (트레이드오프)

- **축 = 상태를 어디 두나 (서버 vs 클라이언트)**
- **세션**: 무효화 쉬움(서버 세션 삭제 = 즉시 차단) ↔ 서버 부담·확장성(서버 여러 대면 세션 공유 필요).
- **JWT**: 확장 쉬움(secret만) ↔ **무효화 어려움**(발급된 토큰은 만료까지 유효. 취소하려면 서버가 블록리스트를 저장해야 하는데 — 그건 stateless 장점 자체를 깸).
- 본인 인출: "JWT 장점 = 상태 저장 안 함인데, 만료 전 취소하려면 반드시 서버 저장 필요 → 장점이 모순에 빠지는 게 문제." (log_18 "JWT 무효화 한계"를 *왜 그 벽이 생기는지*까지 심화.)

## 5. 오늘 DB 척추와의 평행 (한 줄로 꿰기)

- DB의 **정확성↔동시성** 저울과 **같은 구조**: 인증은 **확장성↔무효화** 저울. 둘 다 "양쪽을 동시에 못 가짐".
- 실무 하이브리드(짧은 access JWT + 취소 가능한 refresh token)는 DB에서 **낙관+비관 락 섞어 쓴 것**과 같은 절충.
- **메타 패턴**: "양쪽을 동시에 못 가지니 상황에 맞게 저울추를 놓는다" — 오늘 DB·인증 두 카테고리를 관통.
