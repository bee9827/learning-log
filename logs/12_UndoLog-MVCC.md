### 학습 과정

#### Undo Log란

"Undo" = 되돌린다. 데이터를 수정하기 전의 이전 값을 기록해두는 공간.

**두 가지 목적:**

1. **롤백** — 수정 전 값으로 되돌리기
   - 트랜잭션이 실패하면 Undo Log의 이전 값으로 복원

2. **MVCC** — 다른 트랜잭션에게 이전 버전 보여주기
   - Repeatable Read에서 "내가 처음 읽었을 때의 스냅샷"을 유지하기 위해 사용
   - 다른 트랜잭션이 값을 수정해도 내 트랜잭션은 Undo Log의 이전 버전을 읽음

**readOnly = true일 때 Undo Log를 생략하는 이유:**
수정이 없으면 되돌릴 것도, 이전 버전을 만들 필요도 없기 때문.

---

#### Lost Update 문제

Undo Log가 있어도 동시성 문제가 발생할 수 있다.

```
초기값: 0
A: 0 읽음 → 1로 씀
B: 0 읽음 → 1로 씀 → 커밋
A: 롤백 → Undo Log의 값(0)으로 되돌림
결과: 0 ← B의 커밋이 사라짐!
```

Undo Log는 "A가 쓴 값"을 되돌리는 것이지, B의 커밋을 보존하지 않는다.
→ Undo Log는 동시성 제어 도구가 아님. 별도 동시성 제어가 필요하다.

## 3. 전략 평가

- 효과적이었던 것과 그 이유
  "이전 값으로 되돌리면 B가 덮어씌워지지 않나?"라는 의문을 스스로 제기했고, 시나리오를 통해 직접 확인했다. 의문이 명확했기 때문에 개념이 선명하게 남았다.

- 비효과적이었던 것과 그 이유
  Undo Log의 내부 구조(어떻게 저장되는지, 언제 정리되는지)까지는 다루지 않았다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  의문에서 출발해서 시나리오로 검증하는 방식이 잘 맞았다.

## 4. AI 피드백

Lost Update 시나리오를 스스로 만들어서 "Undo Log가 동시성 제어 도구가 아니다"를 직접 증명했다. 이게 나선형 학습의 핵심 — 개념을 외우는 게 아니라 스스로 반례를 만들어서 경계를 확인하는 것.
