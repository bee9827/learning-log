# 학습 로그 #34

**날짜**: 2026-06-15
**학습 범위**: 테스트의 지향점 — 무엇을, 어느 레벨에서, 얼마나 *진짜로* 검증하나. 격리↔실제성 저울, 테스트↔트랜잭션 용접.

분류: 테스트 (재방문: #02 Controller 테스트, #27 트랜잭션 롤백 테스트 — *직접 인출로 한 줄 꿰기*)

## 1. 지향점 (뿌리 질문)

- 큰 목적: "변경해도 안 깨졌다는 신뢰."
- 실제 결정의 줄기: **"무엇을, 어느 레벨에서, 얼마나 진짜로 검증하나."**

## 2. 저울 — 격리 ↔ 실제성

| 왼쪽 (격리) | 오른쪽 (실제성) |
|---|---|
| Mock(전부 가짜) / JdbcTest(슬라이스) | Spy(거의 진짜) / SpringBootTest(통합) |
| 실제와 멀다 → **빠름·단순** | 무겁다 → **느림** |
| 진짜 동작은 못 본다 | **실제 동작을 검증 → 가짜로 못 잡는 걸 잡는다** |

- **위치 기준**: 검증 대상이 "진짜"를 얼마나 요구하느냐. 상태코드/단순 로직 → 왼쪽(빠르게). 진짜 트랜잭션 경계·진짜 협력 → 오른쪽.

## 3. 본인이 내린 결정들

- **Controller 테스트**: 상태코드(201/400/409)를 `RestAssuredMockMvc`(given/when/then 가독성)로. Service 변경에 흔들리던 예외 케이스를 단위로 **분리** → Controller 중복 제거(테스트 유지보수성).
- **롤백 테스트**: **Spy**로 "진짜 DAO인데 `delete`만 실패" 주입(전부 가짜인 Mock이 아니라). **SpringBootTest**로 진짜 트랜잭션 경계 관찰.

## 4. 테스트 ↔ 트랜잭션 용접점 (throughline)

- `@JdbcTest`가 안 된 진짜 이유: 테스트가 *자기 트랜잭션*을 쥠 → `cancel()`의 `@Transactional`이 거기 **join** → 물리 롤백이 테스트 끝까지 미뤄져 검증이 `CANCELED`를 봄 → 깨짐.
- 즉 **"테스트가 트랜잭션을 쥐면 진짜 경계가 가려진다."** 진짜 경계를 보려면 테스트가 트랜잭션을 안 쥐어야(`@SpringBootTest` + 수동 정리).
- 메커니즘: 트랜잭션 = **커넥션이 쓰레드에 바인딩(ThreadLocal, [[ThreadLocal-커넥션바인딩]] log_09)**되는 방식.

## 5. 메타 — 네 번째 저울

- DB: 정확성↔동시성 / 인증: 확장성↔무효화 / 모델링: 묶기↔나누기 / **테스트: 격리↔실제성.**
- 같은 패턴: "양쪽을 동시에 못 가지니 상황(검증 대상)에 맞게 추를 놓는다." [[ACID지향점-메타복습]] [[인증세션-지향점]] [[모델링-책임의위치]]
